# SMB AI Workflow Audit Kit — 5-Minute Local Demo Script

> Fictional/local demo only. This is not approval to publish, push, contact anyone, collect real business data, inspect private systems, create a waitlist/payment page, deploy anything, spend money, or run a live audit.

Use this walkthrough to review whether the kit is understandable as a fixed-scope SMB workflow audit offer before any external step.

## Demo goal

Show how the kit moves from safe discovery to a packaged audit deliverable:

- start with fictional/redacted-style intake;
- identify repetitive workflows without collecting secrets or exports;
- score opportunities for value, feasibility, and risk;
- produce a concrete report with human review and rollback notes;
- optionally draft a pilot blueprint and acceptance evidence gate;
- stop before outreach, payment, public launch, private-system access, or implementation.

## Setup

From the repository root:

```bash
npm run check
git status --short --branch
```

Expected result: validation passes and the repo is clean after committed work.

## Minute-by-minute walkthrough

### Minute 1 — purpose and guardrails

Open `README.md` and point to:

- fictional sample data only;
- no permission to access tools, inboxes, documents, CRM, accounting systems, or customer records;
- no promises of revenue, savings, compliance, ROI, or guaranteed accuracy;
- human review before any customer-facing or operational use.

Reviewer question: "Can this be understood without touching a real client's systems?"

Expected answer: yes; the current kit can be reviewed entirely with fictional samples and templates.

### Minute 2 — safe intake and discovery

Open:

1. `templates/client-safe-intake-questionnaire.md`
2. `samples/fictional-intake.md`
3. `docs/discovery-call-script.md`

Point out:

- the intake avoids passwords, exports, customer records, bank/payroll/medical/legal data, and account access;
- discovery uses generalized or redacted-style examples;
- a future call would require explicit approval for the named person/business, exact wording, channel, and stop conditions.

Reviewer question: "What gets rejected before the audit starts?"

Expected answer: any request needing secrets, regulated/customer data, system exports, live access, scraping, implementation, payment setup, or customer messaging.

### Minute 3 — scoring and report deliverable

Open:

1. `templates/scoring-rubric.md`
2. `templates/audit-report-template.md`
3. `samples/completed-audit-report.md`

Narrate the value path:

- score boring repetitive workflows first;
- prefer reversible, human-reviewed recommendations;
- include risks, non-goals, rollback/deactivation notes, and approval gates;
- keep the output as decision support, not implementation permission.

Reviewer question: "Would the completed sample feel concrete enough to evaluate as a paid audit deliverable?"

Expected answer: yes if it clarifies top opportunities, risk, next steps, and what not to automate.

### Minute 4 — pilot and evidence gate

Open:

1. `templates/pilot-blueprint-template.md`
2. `samples/fictional-pilot-blueprint.md`
3. `templates/acceptance-scorecard.md`
4. `samples/fictional-acceptance-log.md`

Point out:

- pilot planning is draft-only;
- no automatic customer messages;
- acceptance uses accept/light-edit/reject evidence;
- critical safety flags stop continuation;
- rollback/deactivation is part of the plan.

Reviewer question: "Can a pilot proceed just because the audit sounds promising?"

Expected answer: no; it needs explicit approval, a data-handling plan, human review, stop conditions, and a separate implementation gate.

## Demo pass criteria

The demo passes if a reviewer can answer yes to all of these:

- The offer can be explained without vague "AI transformation" claims.
- The completed fictional report shows a concrete deliverable.
- Safe intake avoids secrets, exports, regulated data, customer records, and account access.
- The pilot blueprint is clearly draft-only and human-reviewed.
- Acceptance evidence and safety flags can stop continuation.
- Externalization paths are approval-gated and limited to exactly one chosen next step.
- Validation passes locally.

## Demo fail criteria

Stop and revise if any are true:

- The kit appears to approve live audits, outreach, publishing, payment pages, or system access.
- The deliverable depends on private business data before written permission and a data-handling plan.
- The report promises ROI, revenue, savings, compliance, or guaranteed accuracy.
- Customer-facing messages could be sent automatically.
- Public copy creates hype or urgency instead of a narrow review offer.
- Skipped checks can be mistaken for passed checks.
