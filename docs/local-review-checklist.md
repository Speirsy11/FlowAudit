# SMB AI Workflow Audit Kit — Local Review Checklist

> Internal review aid only. This checklist is not approval to publish, push, sell, invoice, contact anyone, collect real business data, or create a payment/waitlist page.

Use this before any future external validation step.

## 1. Package completeness

- [ ] README explains the kit, flow, files, and guardrails in plain language.
- [ ] Fictional intake, fictional discovery notes, and completed fictional report all line up around the same sample business.
- [ ] Scoring rubric produces understandable priorities without implying guaranteed ROI.
- [ ] Audit report template includes non-goals, risks, rollback/deactivation steps, approval gates, and human review.
- [ ] Offer options are clearly draft/internal and not a live price sheet.
- [ ] Landing page copy is clearly draft-only and not published.

## 2. Safety and privacy review

- [ ] No real client, customer, company, staff, account, inbox, CRM, accounting, payroll, bank, medical, legal, or regulated data appears anywhere.
- [ ] No secrets, tokens, API keys, credentials, private URLs, phone numbers, email addresses, or personal addresses appear in examples.
- [ ] The docs say not to collect passwords, API keys, bank details, customer records, payroll data, medical data, or legal case details.
- [ ] The docs avoid guarantees around revenue, savings, compliance, accuracy, or ROI.
- [ ] Every proposed automation remains draft-only or human-reviewed unless separately approved.
- [ ] Any real system access would require explicit written permission and a data-handling plan first.

## 3. Offer-readiness review

- [ ] Target customer is narrow enough: owner-operated/local service businesses with low-risk admin workflows.
- [ ] Poor-fit customers and high-stakes workflows are excluded.
- [ ] Deliverable is a fixed-scope decision-support report, not vague AI transformation.
- [ ] Pricing ranges are treated as assumptions to validate, not claims of market acceptance.
- [ ] The first validation goal is learning whether the report is useful enough to pay for, not scaling outreach.

## 4. Approval gates before anything leaves local machine

- [ ] Charlie approves whether this should stay private, move to a private remote, or become a public-safe repo.
- [ ] Charlie approves any landing page, waitlist, payment, scheduling, or analytics setup separately.
- [ ] Charlie approves any outreach/message/call recipient, wording, and channel separately.
- [ ] Charlie approves any real-client data-handling plan before non-fictional examples are collected.
- [ ] Charlie approves any implementation work separately from the audit/report package.

## 5. Suggested next validation sequence

1. Run `npm run check` locally.
2. Read `samples/completed-audit-report.md` end to end as if buying the audit.
3. Review `docs/offer-options.md` for package/pricing clarity.
4. Review `docs/landing-page-draft.md` for claims, safety, and CTA risk.
5. If still promising, choose exactly one approval-gated next step:
   - create a private remote backup;
   - prepare a public-safe repo/pitch page without publishing;
   - or approve one carefully worded validation conversation with a known business owner.

## Current recommendation

The safest revenue move is to keep the kit local, review the completed fictional report, then approve one narrow validation step only if the offer feels concrete enough. Do not publish the landing page or contact anyone before that review.
